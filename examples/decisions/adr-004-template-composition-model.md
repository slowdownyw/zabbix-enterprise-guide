# ADR-004: Модель композиции шаблонов

Статус: Предложен  
Дата: ГГГГ-ММ-ДД  
Владелец: Архитектор мониторинга  
Принимают решение: Владелец мониторинга, администратор Zabbix, владельцы приложений, владельцы платформ  
Дата пересмотра: после пилотного развёртывания шаблонов  
Область: глобальный дизайн шаблонов

---

## Что решаем

Как должны быть устроены шаблоны Zabbix, чтобы оставаться переиспользуемыми, понятными и сопровождаемыми?

---

## Контекст

Типичный антипаттерн в Zabbix — монолитный шаблон:

```text
Шаблон: Plant Everything for 1C Server
```

Он содержит метрики ОС, проверки процессов, приложений, БД, синтетику, теги сервисов и локальные исключения. Работает на первом хосте и становится неуправляемым на десятом.

Зрелый дизайн разделяет ответственности:

```text
Base → Role → Application → Synthetic
```

Это **композиция**, а не объектно-ориентированное наследование. Цель — не строить хитрую иерархию шаблонов. Цель — прикрепить правильные обязанности мониторинга к правильным переиспользуемым единицам.

---

## Критерии выбора

1. Исключить дублирование метрик.
2. Сохранить переиспользуемость мониторинга ОС.
3. Отделить логику приложения от логики платформы.
4. Сделать изменения шаблонов безопаснее.
5. Поддержать поэтапный пилот / развёртывание по слоям.
6. Сделать теги событий и runbook'и предсказуемыми.
7. Избежать ситуации, когда каждая команда изобретает собственные базовые проверки.

---

## Рассмотренные варианты

### Вариант A — Использовать только вендорские/дефолтные шаблоны

Использовать официальные и community-шаблоны с минимальной кастомизацией.

**Выбирать когда:**

- ранняя стадия discovery;
- лаборатория или быстрый baseline;
- у команды нет времени разрабатывать кастомные шаблоны;
- официальный шаблон хорошо покрывает задачу.

**Избегать когда:**

- высокий alert noise;
- нужна сервисная семантика;
- дашборды/SLA требуют согласованных тегов;
- нужны enterprise-специфичные пороги и runbook'и.

**Плюсы:**

- быстрый старт;
- upstream-поддержка шаблонов;
- меньше кастомного кода.

**Минусы:**

- непоследовательные теги и severity;
- слишком общие для сервисных операций;
- не отражают локальное владение/SLA/runbook'и;
- сложно стандартизировать на разнородных системах.

---

### Вариант B — Один кастомный монолитный шаблон на систему

Создавать шаблоны типа `Plant: 1C Server`, `Plant: Exchange`, `Plant: MSSQL`, каждый со всеми метриками для данной системы.

**Выбирать когда:**

- небольшая среда;
- одна команда всем владеет;
- у системы мало компонентов;
- скорость важнее долгосрочного сопровождения.

**Избегать когда:**

- шаблоны будут переиспользоваться в разных ролях;
- ответственность за ОС/БД/приложение разделена;
- несколько команд участвуют;
- важны сервисные дашборды и LLD.

**Плюсы:**

- простая ментальная модель поначалу;
- меньше шаблонов подключать;
- простое демо.

**Минусы:**

- дублирование метрик ОС;
- сложно менять один слой независимо;
- триггеры и теги становятся непоследовательными;
- не подходит для смешанных ролей;
- со временем создаёт «кладбище шаблонов».

---

### Вариант C — Слоистая модель композиции

Шаблоны сгруппированы по ответственности:

```text
Common Policy шаблоны
Base OS/network/storage шаблоны
Role шаблоны
Application шаблоны
Synthetic/business шаблоны
```

Хосты получают несколько шаблонов, которые вместе описывают их роль.

**Выбирать когда:**

- средняя или крупная среда;
- множество хостов разделяют базовый мониторинг;
- прикладные роли отличаются;
- дашборды и алертинг зависят от согласованных тегов;
- командам нужны границы ответственности.

**Избегать когда:**

- команда не может управлять несколькими связанными шаблонами;
- Zabbix используется только для короткого PoC;
- нет процесса ревью шаблонов.

**Плюсы:**

- переиспользуемость;
- проще тестировать;
- снижает дублирование;
- поддерживает сервис-специфичные роли;
- согласуется со слоистой tag-моделью;
- удобнее поэтапное развёртывание.

**Минусы:**

- требует соглашений об именовании;
- требует документирования;
- список шаблонов хоста становится длиннее;
- нужно понимать зависимости шаблонов.

---

### Вариант D — Шаблоны, генерируемые из кода

Шаблоны генерируются из YAML/JSON-моделей или управляются через Ansible/API-workflow.

**Выбирать когда:**

- у команды сильные навыки автоматизации;
- изменения Zabbix должны проходить code review;
- много похожих сервисов/хостов;
- требуется drift detection.

**Избегать когда:**

- модель шаблонов ещё не стабильна;
- команда не имеет Git/CI-дисциплины;
- UI-эксперименты ещё часты;
- сгенерированный результат нельзя отлаживать операторами.

**Плюсы:**

- воспроизводимость;
- возможность ревью;
- обеспечивает соблюдение стандартов;
- хорошо масштабируется.

**Минусы:**

- больше инженерных накладных расходов;
- опасно при неверной модели;
- может оттолкнуть UI-ориентированных операторов;
- не снимает необходимости в дизайне.

---

## Решение

Рекомендуемый baseline:

> Используйте **слоистую модель композиции** для всех новых Plant-шаблонов. Не создавайте монолитные «всё-в-одном» шаблоны, кроме как временных обёрток при миграции.

Классы шаблонов:

```text
Common Policy
Base
Role
Application
Synthetic
```

Пример для хоста с приложением 1С:

```text
Хост: 1c-app-warehouse-01
  ├─ Plant: Common Policy
  ├─ Plant: Windows Base
  ├─ Plant: Windows Services Base
  ├─ Plant: Role - 1C App Node
  ├─ Plant: 1C App Server
  ├─ Plant: HASP Client
  └─ Plant: Synthetic - 1C Web Login
```

---

## Определения классов шаблонов

### Common Policy

Назначение:

- общие макросы;
- общие теги;
- соглашения об именовании;
- лёгкие policy-объекты.

Правила:

- никакого тяжёлого сбора items;
- никаких прикладных проверок;
- никаких скрытых побочных эффектов.

### Base-шаблоны

Назначение:

- baseline ОС/платформы;
- CPU, память, диски, LLD файловых систем;
- LLD сетевых интерфейсов;
- health агента;
- проверки базовых сервисов.

Примеры:

```text
Plant: Windows Base
Plant: Linux Base
Plant: Network SNMP Base
Plant: Storage Base
```

### Role-шаблоны

Назначение:

- проверки роли хоста;
- предварительные требования;
- специфичные для роли процессы/порты;
- метаданные роли.

Примеры:

```text
Plant: Role - 1C App Node
Plant: Role - MSSQL Server
Plant: Role - Exchange DAG Member
Plant: Role - Veeam Repository
```

### Application-шаблоны

Назначение:

- специфичные для приложения items/triggers;
- предметно-ориентированный LLD;
- runbook'и приложения;
- теги component/scope/impact.

Примеры:

```text
Plant: 1C App Server
Plant: MSSQL for 1C
Plant: Exchange DAG
Plant: Veeam Backup RPO
```

### Synthetic-шаблоны

Назначение:

- бизнес-ориентированные проверки;
- HTTP-проверки;
- end-to-end транзакции;
- доступность сервиса с точки зрения пользователя.

Примеры:

```text
Plant: Synthetic - 1C Web Login
Plant: Synthetic - OWA Login
Plant: Synthetic - SMTP Send Receive
```

---

## Последствия

### Положительные

- О шаблонах проще рассуждать.
- Команды могут владеть своим слоем.
- Метрики ОС не дублируются в application-шаблонах.
- Тегирование LLD/prototype можно стандартизировать.
- Пилотное развёртывание может применять по одному слою за раз.
- Дашборды могут опираться на согласованные теги component/scope.

### Негативные / компромиссы

- В конфигурации хоста появляется больше шаблонов.
- Нужно документировать, какие шаблоны относятся к каждой роли.
- Неверная слоистость может создать пробелы или дублирующиеся алерты.
- Операторам нужно обучение по модели композиции.

### Операционные последствия

- Новые шаблоны должны объявлять свой класс.
- Ревью шаблонов должно проверять дублирование и размещение тегов.
- Подключение хоста должно использовать правильный набор шаблонов.
- Изменения шаблонов должны тестироваться на представительном хосте роли.

---

## Заметки по реализации

Рекомендуемый заголовок шаблона в документации:

```text
Название шаблона: Plant: MSSQL for 1C
Класс:            Application
Зависит от:       Plant: Windows Base, Plant: Role - MSSQL Server
Создаёт теги:     component=mssql, scope=performance|availability|capacity
Runbook'и:        rb-mssql-blocking, rb-mssql-backup-age
Статус:           Design draft / Lab-tested / Production-tested
```

Контрольные вопросы при ревью:

- Дублирует ли шаблон items из Base?
- Содержит ли он только прикладные проверки?
- Тегированы ли LLD prototype?
- Связаны ли триггеры High/Disaster с runbook'ами?
- Задокументированы ли макросы?
- Нужны ли value maps?
- Зависит ли шаблон от Windows/Linux-пути или локали?

---

## Исключения

Временные шаблоны-обёртки для миграции допустимы:

```text
Plant: Legacy Bundle - 1C App Host
```

Но они обязаны:

- быть помечены как временные;
- иметь владельца;
- иметь план декомпозиции;
- не использоваться для новых сервисов.

---

## Критерии приёмки

- [ ] Классы шаблонов задокументированы.
- [ ] Пилотный хост использует несколько скомпонованных шаблонов.
- [ ] Нет дублирования метрик ОС в application-шаблоне.
- [ ] Теги LLD prototype присутствуют там, где применимо.
- [ ] Триггеры High/Disaster имеют тег runbook или URL.
- [ ] Список шаблонов для каждой стандартной роли задокументирован.
- [ ] Checklist ревью шаблонов существует.

---

## Триггеры пересмотра

Пересмотреть этот ADR, если:

- команды не могут работать со скомпонованными шаблонами;
- количество шаблонов становится неуправляемым;
- официальные шаблоны Zabbix значительно изменились;
- принят workflow генерации/IaC-шаблонов;
- инцидент показал нечёткость границ ответственности шаблонов.
