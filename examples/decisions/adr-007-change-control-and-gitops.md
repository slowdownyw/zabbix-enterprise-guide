# ADR-007: Change control и GitOps-политика мониторинга

Статус: Предложен  
Дата: ГГГГ-ММ-ДД  
Владелец: Владелец мониторинга  
Принимают решение: Архитектор мониторинга, администратор Zabbix, руководитель операций, ИБ, владельцы сервисов  
Дата пересмотра: после первой волны развёртывания  
Область: конфигурация продакшн Zabbix

---

## Что решаем

Как должны контролироваться, проверяться, экспортироваться и аудироваться изменения продакшн-конфигурации мониторинга?

---

## Контекст

Zabbix хранит конфигурацию в БД. В отличие от Kubernetes-стиля GitOps, БД является живым источником рантайм-конфигурации. Инженеры могут менять шаблоны, триггеры, actions, media types, host groups, теги и дашборды через UI.

Это создаёт противоречие:

- UI-изменения быстрые и практичные;
- Git лучше для ревью и истории;
- полное декларативное состояние — сложно;
- ручные экспорты часто устаревают;
- бесконтрольные UI-изменения создают drift;
- слишком раннее запрещение UI блокирует операции.

Цель — не идеологический GitOps. Цель — контролируемое, аудируемое, откатываемое изменение.

---

## Критерии выбора

1. Предотвратить случайные сбои в продакшне.
2. Хранить историю изменений мониторинга.
3. Разрешить экстренные исправления.
4. Избежать drift конфигурации.
5. Поддержать ревью шаблонов.
6. Сохранить операционную скорость.
7. Сделать владение видимым.

---

## Рассмотренные варианты

### Вариант A — Только UI-изменения

Все изменения вносятся напрямую в UI Zabbix.

**Выбирать когда:**

- маленькая команда;
- ранний lab/пилот;
- нет навыков Git/CI;
- скорость важнее аудируемости.

**Избегать когда:**

- продакшн-мониторинг критичен;
- несколько инженеров могут редактировать шаблоны;
- важен регуляторный/ИБ-аудит;
- нужен откат;
- много шаблонов/actions.

**Плюсы:**

- быстрее всего;
- удобно для администраторов Zabbix;
- нет оверхеда инструментария.

**Минусы:**

- слабый audit trail;
- сложный откат;
- скрытый drift;
- изменения зависят от памяти конкретного человека;
- сложно проверить.

---

### Вариант B — Ручной экспорт в Git после изменений

Инженеры меняют Zabbix в UI и периодически экспортируют шаблоны/конфигурацию в Git.

**Выбирать когда:**

- организация начинает путь к зрелости;
- полная автоматизация не готова;
- шаблоны ещё развиваются;
- команда хотя бы способна поддерживать дисциплину экспорта.

**Избегать когда:**

- экспорты не обязательны;
- много людей независимо меняют UI;
- история Git должна отражать каждое продакшн-изменение.

**Плюсы:**

- простое улучшение по сравнению с UI-only;
- создаёт исторические снапшоты;
- нет сложной автоматизации.

**Минусы:**

- экспорты можно забыть;
- Git может не соответствовать текущему Zabbix;
- нет реального согласования до изменения;
- откат всё равно ручной.

---

### Вариант C — Плановый экспорт и обнаружение drift

Scheduled-задание экспортирует выбранную конфигурацию Zabbix в Git и автоматически коммитит изменения. Ручные UI-изменения становятся видны как diff.

**Выбирать когда:**

- UI-изменения всё ещё разрешены;
- нужна видимость drift;
- команда хочет аудит без полного IaC;
- процесс продакшна взрослеет.

**Избегать когда:**

- экспортируемая конфигурация содержит секреты;
- никто не просматривает diff;
- формат экспорта слишком шумный, чтобы быть полезным.

**Плюсы:**

- Git отражает реальные снапшоты состояния;
- скрытые UI-изменения становятся видны;
- практичный промежуточный уровень зрелости;
- низкое трение.

**Минусы:**

- не предотвращает плохие изменения;
- автоматические diff могут быть шумными;
- секреты/чувствительные данные должны фильтроваться;
- откат всё ещё требует процедуры.

---

### Вариант D — PR-based изменения со скриптовым импортом/apply

Изменения предлагаются в Git, проверяются, затем применяются через скрипты/API/Ansible/import-пайплайны.

**Выбирать когда:**

- модель шаблонов стабильна;
- команда может работать с CI/CD;
- изменения требуют ревью;
- продакшн-мониторинг критичен;
- важна воспроизводимость.

**Избегать когда:**

- объектная модель Zabbix ещё исследуется;
- команда не готова к Git-workflow;
- экстренные операции требуют частых UI-исправлений;
- инструментарий не покрывает нужные объекты.

**Плюсы:**

- проверяемо;
- воспроизводимо;
- лучший сценарий отката;
- поддерживает соблюдение стандартов;
- ближе к настоящему IaC.

**Минусы:**

- сложность инструментария;
- частичное покрытие типов объектов Zabbix;
- может замедлять легитимные исправления;
- требует навыков и дисциплины.

---

### Вариант E — Строгий декларативный IaC

Конфигурация Zabbix полностью управляется как код, UI-изменения запрещены или перезаписываются.

**Выбирать когда:**

- зрелая команда platform engineering;
- стабильная объектная модель;
- строгие compliance-требования;
- покрытие инструментарием доказано;
- Zabbix трактуется как управляемая платформа.

**Избегать когда:**

- большинство работы всё ещё в UI;
- шаблоны незрелые;
- организация не может поддерживать пайплайн;
- покрытие провайдера/модуля Zabbix недостаточно.

**Плюсы:**

- сильнейший контроль;
- воспроизводимость;
- масштабируемый governance.

**Минусы:**

- высокая стоимость внедрения;
- не все объекты Zabbix легко управляются декларативно;
- риск «бороться с продуктом»;
- может стать проектом внутри проекта.

---

## Решение

Рекомендуемый baseline:

> Начните с **планового экспорта + видимости drift** для всего продакшн Zabbix, и требуйте **PR/ревью для Plant-шаблонов, tag schema, severity model, alert actions и определений дашбордов/отчётов**. Переходите к scripted apply/Ansible только после стабилизации модели.

Это прагматичный средний путь:

```text
UI разрешён, но не невидим.
Git используется, но без претензий на идеальный GitOps.
Критические стандарты проверяются до развёртывания.
```

---

## Классы изменений

| Класс изменения | Примеры | Требуемый контроль |
|---|---|---|
| Экстренное операционное исправление | отключить сломанный триггер, временный порог | UI разрешён, тикет/комментарий обязателен, экспорт фиксирует drift |
| Логика шаблона | items, triggers, LLD, макросы | PR/ревью до развёртывания |
| Tag schema | обязательные теги, допустимые значения | PR + согласование владельца |
| Severity model | сопоставление severity, логика эскалации | согласование Владельца мониторинга/руководителя NOC |
| Alert actions | маршрутизация, media, эскалация | контролируемое изменение + тест |
| Дашборды/отчёты | CIO/NOC/сервисные представления | ревью владельца |
| Users/RBAC/media types | права, каналы | ревью ИБ/владельца |

## Последствия

### Положительные

- Drift становится видимым.
- UI остаётся применимым во время перехода.
- Критические design-артефакты проверяются.
- Команда избегает оверинжиниринга полного GitOps слишком рано.
- Будущая автоматизация Ansible/API остаётся возможной.

### Негативные / компромиссы

- UI-изменения всё ещё могут происходить до ревью.
- Экспортные diff могут быть шумными.
- Кто-то должен владеть ревью автоматических коммитов.
- Секреты не должны попадать в коммиты.
- Процесс отката остаётся частично ручным.

### Операционные последствия

- Создать выделенный read-only/export API-токен.
- Запланировать экспорт в Git.
- Определить фильтрацию секретов.
- Определить, кто проверяет drift-коммиты.
- Определить процесс экстренных изменений.
- Добавить «источник изменения» в постмортемы, если изменение мониторинга было причиной.

## Заметки по реализации

Минимальная структура репозитория:

```text
zabbix-config/
  exports/
    templates/
    media-types/
    actions/
    host-groups/
  standards/
    tag-schema.yaml
    severity-policy.md
    template-review-checklist.md
  scripts/
    export-zabbix-config.py
```

Минимальный процесс:

```text
1. Плановый экспорт запускается ежедневно или ежечасно.
2. Экспортная задача открывает коммит при наличии diff.
3. Владелец мониторинга еженедельно проверяет значимые diff.
4. Изменения Plant-шаблонов проходят через PR.
5. Экстренные UI-изменения требуют тикета/комментария.
6. Продакшн-импорт/apply выполняется именованной ролью.
```

Безопасность секретов:

- не экспортировать API-токены в Git;
- очищать секреты media types;
- не коммитить реальные credentials, webhook-секреты, PSK-значения;
- держать `.env`, экспортированные CSV-инвентаризации и сырые API-дампы вне публичного репозитория.

---

## Исключения

UI-only изменения допустимы для:

- экстренного устранения инцидента;
- временной настройки в рамках maintenance;
- lab/pilot-объектов не в продакшне;
- исследовательской работы перед формализацией шаблона.

Но любое UI-only продакшн-изменение, живущее дольше согласованного временного окна, должно быть зафиксировано в Git или формализовано через процесс изменений.

---

## Критерии приёмки

- [ ] Процесс экспорта существует.
- [ ] Git-репозиторий хранит очищенные экспорты/стандарты.
- [ ] Для изменений Plant-шаблонов есть путь ревью.
- [ ] Процесс экстренных UI-изменений существует.
- [ ] Определён ритм проверки drift.
- [ ] Секреты исключены из коммитов.
- [ ] Хотя бы один сценарий отката задокументирован.

---

## Триггеры пересмотра

Пересмотреть этот ADR, если:

- drift участился;
- UI-изменения вызвали инциденты;
- экспортные diff нечитаемы;
- команда перешла на Ansible/community.zabbix;
- compliance требует более строгого change control;
- количество шаблонов/actions значительно выросло.
