# ADR-001: Модель развёртывания Zabbix

Статус: Предложен  
Дата: ГГГГ-ММ-ДД  
Владелец: Архитектор мониторинга  
Принимают решение: Владелец мониторинга, владелец платформы, владелец БД, сетевой инженер, ИБ  
Дата пересмотра: после пилота / до продакшн-развёртывания  
Область: глобально

---

## Что решаем

Какую модель развёртывания Zabbix использовать для целевой платформы мониторинга?

---

## Контекст

Платформа мониторинга превращается в операционный control plane, а не просто инструмент одного инженера. Она будет собирать метрики с инфраструктуры, сетевого оборудования, приложений, баз данных, систем резервного копирования, синтетических проверок и, возможно, изолированных / регулируемых сегментов.

Модель развёртывания должна поддерживать:

- рост числа хостов, items, триггеров и объёма истории;
- чёткие сетевые границы;
- предсказуемое размещение прокси;
- резервное копирование и восстановление;
- security review;
- будущее масштабирование без переработки всей архитектуры;
- разделение между логикой мониторинга и бизнес-дашбордами.

Zabbix server — центральный процесс, выполняющий polling/trapping, вычисление триггеров и отправку уведомлений. Прокси могут собирать данные от имени сервера, буферировать их локально и помогать распределить нагрузку. Размер БД сильно зависит от количества items, интервалов обновления и retention.

---

## Критерии выбора

1. Надёжность мониторинга при нестабильной сети.
2. Производительность БД и рост хранилища.
3. Сегментация между площадками, DMZ, OT и корпоративной сетью.
4. Операционная простота при первом развёртывании.
5. Возможность масштабирования без полной переработки.
6. Ожидания по резервному копированию / восстановлению и disaster recovery.
7. Навыки команды и модель поддержки.

---

## Рассмотренные варианты

### Вариант A — Единый all-in-one Zabbix-сервер

Zabbix server, frontend и БД работают на одной VM / одном сервере.

**Выбирать когда:**

- лаборатория, PoC или небольшая среда;
- ожидаемый объём метрик невысок;
- строгих требований к сегментации нет;
- цель — обучение или краткосрочная валидация;
- простои мониторинга на техническое обслуживание допустимы.

**Избегать когда:**

- мониторинг критически важен для бизнеса;
- есть несколько площадок или изолированных сегментов;
- ожидается значительный рост базы данных;
- строгие требования к backup/restore;
- командам нужна раздельная ответственность за БД / платформу / ИБ.

**Плюсы:**

- быстрее всего разворачивается;
- проще всего troubleshoot на старте;
- минимальный footprint инфраструктуры;
- хорошо для лабораторных и пилотных экспериментов.

**Минусы:**

- БД и сервер конкурируют за CPU, память и I/O;
- сложно масштабировать без переделки;
- любое техническое обслуживание затрагивает весь стек;
- плохо подходит для сегментированных enterprise-сред;
- слабая основа для будущего HA.

---

### Вариант B — Zabbix server/frontend и БД на разных узлах

Zabbix server/frontend работает на одном узле или паре узлов; PostgreSQL/MySQL — на отдельном сервере БД.

**Выбирать когда:**

- среднего размера окружение;
- производительность БД важна;
- команда БД / платформы должна владеть backup, тюнингом, репликацией, хранилищем;
- нужен чистый путь роста, но full HA в первый день не требуется;
- большинство проверок собирается напрямую или через небольшое количество прокси.

**Избегать когда:**

- сеть между Zabbix server и БД нестабильна;
- команда не может эксплуатировать отдельную платформу БД;
- среда очень маленькая и простота важнее масштабирования.

**Плюсы:**

- чёткое разграничение ответственности;
- лучший тюнинг БД и контроль хранилища;
- более простой дизайн backup/restore;
- легче мигрировать в будущем на HA-базу данных;
- более реалистичный enterprise baseline.

**Минусы:**

- больше компонентов инфраструктуры;
- связность с БД становится критичной;
- требуется операционная ответственность платформы/БД;
- без прокси недостаточно для удалённых площадок или сегментированных сетей.

---

### Вариант C — Центральный Zabbix server + отдельная БД + прокси по площадкам / сегментам

Zabbix server и БД централизованы; Zabbix-прокси собирают данные с удалённых площадок, DMZ, сетевых зон или OT-сегментов.

**Выбирать когда:**

- есть несколько площадок или сетевых сегментов;
- удалённые каналы ненадёжны;
- firewall-политика должна быть простой;
- polling-нагрузку нужно распределить;
- OT/SCADA или DMZ не должны мониториться напрямую с центрального сервера;
- важна локальная буферизация при падении WAN.

**Избегать когда:**

- среда крошечная;
- владелец прокси не определён;
- нет процесса обновления прокси и резервного копирования;
- команда хочет «просто один сервер» и принимает операционные ограничения.

**Плюсы:**

- хороший enterprise-default;
- снижает нагрузку центрального polling;
- поддерживает сетевую сегментацию;
- прокси буферируют данные при временной потере связи;
- чистая модель ответственности по площадкам / зонам;
- проще firewall-правила: прокси → сервер вместо сервер → всё.

**Минусы:**

- больше движущихся частей;
- мониторинг самих прокси становится обязательным;
- развёртывание шаблонов должно учитывать назначение прокси;
- неправильное размещение прокси создаёт технический и организационный долг.

---

### Вариант D — HA Zabbix server + HA / managed database + прокси

Zabbix server развёрнут с высокой доступностью, БД — реплицирована / managed / HA, прокси используются по площадке / сегменту.

**Выбирать когда:**

- мониторинг является mission-critical операционной системой;
- NOC зависит от Zabbix в режиме 24/7;
- недоступность мониторинга сама по себе должна трактоваться как инцидент;
- есть строгие требования RTO/RPO для мониторинга;
- большой масштаб или регуляторные требования;
- организация умеет эксплуатировать HA-стек.

**Избегать когда:**

- команда не может надёжно эксплуатировать HA-базу данных;
- процесс мониторинга ещё незрелый и алерты зашумлены;
- проект на ранней стадии discovery/пилота;
- HA скрыл бы нерешённые проблемы дизайна под дополнительной инфраструктурой.

**Плюсы:**

- лучшая операционная устойчивость;
- поддерживает критичные NOC-сценарии;
- платформа мониторинга переживает отказ компонента;
- подходит для зрелой enterprise-эксплуатации.

**Минусы:**

- наибольшая сложность;
- более высокая стоимость;
- требует дисциплины БД и платформы;
- HA не исправляет плохие шаблоны, теги или алертинг;
- тестирование failover становится частью регулярных операций.

---

## Решение

Рекомендуемый baseline для среднего и крупного enterprise-окружения:

> Используйте **Вариант C** как целевую модель по умолчанию: центральный Zabbix server/frontend, отдельная БД и Zabbix-прокси на каждой крупной площадке или сетевом/сегменте безопасности. Сохраните **Вариант D** как цель для сред, где мониторинг формально является mission-critical и команда способна отвечать за HA.

Для небольших / лабораторных сред Вариант A допустим только как временный этап.

---

## Последствия

### Положительные

- Размещение прокси становится частью архитектуры, а не запоздалой мыслью.
- БД можно размерить и эксплуатировать независимо.
- Сетевые и ИБ-команды получают чёткую модель границ.
- Удалённые площадки и изолированные сегменты можно подключать постепенно.
- Будущий апгрейд до HA возможен без переработки всех проверок.

### Негативные / компромиссы

- Жизненный цикл прокси должен быть в чьей-то ответственности: патчинг, мониторинг, дисковое пространство, резервное копирование.
- Troubleshooting теперь включает сервер, БД, прокси, агент и сетевой путь.
- Неправильное размещение прокси создаёт операционный долг.
- Часть команд поначалу может сопротивляться «ещё одному компоненту».

### Операционные последствия

- У каждого хоста должен быть определён путь сбора данных: напрямую через сервер или через прокси.
- Здоровье прокси должно быть частью self-monitoring.
- Firewall-правила следует задокументировать по зонам прокси.
- Checklist подключения хостов должен включать назначение прокси.
- Capacity planning должен охватывать центральный сервер, БД и прокси отдельно.

---

## Заметки по реализации

1. Начните с диаграммы развёртывания в документации проекта.
2. Определите зоны / площадки и назначьте предполагаемые прокси.
3. На пилоте разверните минимальный набор прокси, достаточный для валидации модели.
4. Не подключайте OT/SCADA-мониторинг напрямую к центральным проверкам без согласования с ИБ/АСУ ТП.
5. Определите соглашение об именовании прокси:

```text
zbx-prx-<площадка>-<сегмент>-<nn>
```

Примеры:

```text
zbx-prx-dc-main-it-01
zbx-prx-dmz-corp-01
zbx-prx-otdmz-plant1-01
```

6. Добавьте теги / метаданные прокси там, где это полезно:

```text
location=dc-main
segment=it
role=zabbix-proxy
owner=monitoring
```

---

## Исключения

Прямой polling с сервера допустим, когда:

- хост находится в том же доверенном серверном сегменте;
- прокси ещё не существует и scope пилота ограничен;
- тип item требует специфической центральной проверки;
- исключение задокументировано и проверено до массового развёртывания.

Все долгосрочные исключения должны иметь владельца и дату пересмотра.

---

## Критерии приёмки

- [ ] Диаграмма развёртывания существует и согласована.
- [ ] Размещение БД задокументировано.
- [ ] Топология прокси задокументирована.
- [ ] Пилотные хосты собирают данные через предполагаемые пути.
- [ ] Мониторинг здоровья прокси включён.
- [ ] Firewall-правила соответствуют выбранной модели.
- [ ] Подход к backup/restore БД Zabbix задокументирован.
- [ ] Sizing baseline включает предположения по метрикам / items / NVPS и retention.

---

## Триггеры пересмотра

Пересмотреть этот ADR, если:

- число отслеживаемых метрик выросло более чем в 2 раза;
- добавились новые площадки или зоны OT/DMZ;
- мониторинг стал mission-critical в режиме 24/7;
- производительность БД стала узким местом;
- инцидент показал неправильное размещение прокси или недостатки центральной архитектуры.
