# ADR-002: Режим работы агента и baseline шифрования

Статус: Предложен  
Дата: ГГГГ-ММ-ДД  
Владелец: Архитектор мониторинга  
Принимают решение: Владелец мониторинга, сетевой инженер, ИБ, администратор Zabbix, владелец платформы  
Дата пересмотра: после пилота / после security review  
Область: глобально, с документированными исключениями

---

## Что решаем

Какой режим работы Zabbix-агента и политика шифрования должны быть стандартом для продакшн-хостов?

---

## Контекст

Zabbix поддерживает пассивные и активные проверки агента.

При пассивной проверке Zabbix server или прокси сам запрашивает данные у агента. При активной — агент сам подключается к серверу или прокси, получает список проверок и отправляет данные.

Выбор влияет на:

- направление firewall-правил;
- совместимость с NAT и DMZ;
- топологию прокси;
- модель troubleshooting;
- autoregistration;
- уровень безопасности;
- стандарты для Windows/Linux хостов.

Шифрование — отдельное, но связанное решение. Zabbix поддерживает TLS-шифрование между компонентами с помощью сертификатов или PSK (pre-shared keys). Для большинства enterprise-развёртываний PSK — прагматичный baseline; TLS на сертификатах может требоваться в средах с повышенными требованиями ИБ.

---

## Критерии выбора

1. Безопасность продакшн-мониторинга «из коробки».
2. Простая модель firewall-правил.
3. Масштабируемое подключение хостов.
4. Совместимость с прокси и удалёнными площадками.
5. Операционная понятность troubleshooting.
6. Поддержка baseline для Windows и Linux.
7. Исключить ситуацию «у каждого хоста своя схема».

---

## Рассмотренные варианты

### Вариант A — Пассивные проверки по умолчанию, без шифрования

Zabbix server/прокси подключается к каждому агенту на TCP/10050. Шифрование не используется.

**Выбирать когда:**

- лабораторная среда;
- очень маленькая доверенная сеть;
- временный troubleshooting;
- legacy-парк, где внедрение шифрования заблокировано.

**Избегать когда:**

- продакшн-среда;
- DMZ или удалённые площадки;
- baseline безопасности требует шифрования трафика мониторинга;
- многие хосты за firewall / NAT;
- важны autoregistration и масштабируемое подключение.

**Плюсы:**

- проще всего понять;
- простой troubleshooting через `zabbix_get`;
- работает со многими legacy-примерами.

**Минусы:**

- слабый уровень безопасности;
- сервер / прокси должен дотягиваться до каждого хоста;
- firewall-правила становятся широкими и шумными;
- плохо подходит для сегментированных сетей;
- не масштабируется операционно.

---

### Вариант B — Пассивные проверки по умолчанию с TLS/PSK

Zabbix server / прокси подключается к агентам, но трафик зашифрован.

**Выбирать когда:**

- необходим центральный polling;
- ИБ требует шифрования трафика;
- хосты доступны с прокси;
- команда хорошо контролирует firewall-правила;
- важна простота troubleshooting пассивных проверок.

**Избегать когда:**

- хосты за NAT или в сетях только с исходящим доступом;
- большой распределённый парк;
- firewall-политика строго предпочитает исходящие соединения с мониторируемых хостов.

**Плюсы:**

- сохраняет модель пассивного troubleshooting;
- улучшает безопасность;
- полезен для отдельных серверных сегментов.

**Минусы:**

- по-прежнему требует входящего доступа к каждому агенту;
- необходимо управление PSK / сертификатами;
- неудобно для мобильных / удалённых хостов;
- не идеален как универсальный стандарт.

---

### Вариант C — Активные проверки по умолчанию с TLS/PSK

Агенты сами подключаются к назначенному серверу / прокси, получают список активных проверок и отправляют данные. PSK используется как метод шифрования по умолчанию, если не требуется TLS на сертификатах.

**Выбирать когда:**

- enterprise / multi-site среда;
- хосты должны инициировать исходящие соединения;
- прокси используются по площадкам / сегментам;
- планируется autoregistration;
- важна простота firewall-правил;
- развёртывание Windows/Linux должно быть стандартизировано.

**Избегать когда:**

- тип item требует пассивных проверок;
- нет процесса управления PSK identity / ключами;
- legacy-скрипты завязаны на пассивное поведение агента;
- для отдельных проверок нужен точный запрос/ответ в реальном времени.

**Плюсы:**

- хороший стандарт для сегментированных сетей;
- упрощённое направление firewall: хост → прокси/сервер;
- соответствует архитектуре с прокси;
- хорошо работает с автоматическим подключением;
- шифрование из коробки.

**Минусы:**

- не все типы items / сценарии работают с активными проверками;
- troubleshooting требует понимания очереди активных проверок и обновления конфига;
- жизненный цикл PSK нужно контролировать;
- неправильный `ServerActive` молча ломает поток данных.

---

### Вариант D — Смешанная политика по сегменту и типу item

Активный+PSK — стандарт, но пассивный+TLS/PSK допускается для отдельных сегментов или типов item.

**Выбирать когда:**

- в продакшн-парке смешанные технические ограничения;
- сетевые устройства, appliance или legacy-системы требуют исключений;
- часть проверок должна инициироваться прокси / сервером;
- политика DMZ/OT/ИБ отличается от корпоративной.

**Избегать когда:**

- исключения не документируются;
- каждая команда придумывает собственную схему;
- нет владельца стандартов подключения хостов.

**Плюсы:**

- реалистичная enterprise-модель;
- обрабатывает крайние случаи без блокировки стандартизации;
- позволяет постепенную миграцию.

**Минусы:**

- требует дисциплины документирования;
- checklist подключения должен быть точным;
- команды поддержки должны знать, какая схема применяется к каждому хосту.

---

## Решение

Рекомендуемый baseline:

> Используйте **активные проверки Zabbix agent / agent 2 с TLS/PSK** как стандарт для продакшн-хостов Windows и Linux. Пассивные проверки допускаются только как документированные исключения или для конкретных типов items / сегментов, где активные проверки неприменимы.

Для небольших лабораторий пассивные незашифрованные проверки могут использоваться временно, но не должны представляться как продакшн-стандарт.

---

## Последствия

### Положительные

- Чёткий стандарт для новых хостов.
- Firewall-политику проще стандартизировать.
- Прокси становятся естественными точками сбора.
- Подключение хостов можно автоматизировать.
- Baseline безопасности сильнее, чем при незашифрованном мониторинге.

### Негативные / компромиссы

- Необходимо спроектировать управление PSK identity и ключами.
- Troubleshooting активных проверок требует других привычек.
- Часть legacy-шаблонов может потребовать корректировки.
- Пассивные исключения не должны превращаться в неучтённый sprawl.

### Операционные последствия

- Baseline развёртывания агентов должен задавать `ServerActive`, стратегию `Hostname`/metadata и настройки TLS.
- Правила autoregistration должны проходить проверку ИБ.
- Checklist подключения хостов должен включать режим агента и назначение прокси.
- Команда мониторинга должна задокументировать troubleshooting «активные проверки агента не приходят».

---

## Заметки по реализации

Рекомендуемые параметры baseline:

```text
Тип агента:     Zabbix agent 2 там, где поддерживается
Тип item:       Zabbix agent (active)
Шифрование:     TLS PSK
Endpoint:       ближайший Zabbix-прокси
Пассив:         только как исключение
```

Пример текста политики:

```text
Все новые продакшн-хосты Windows и Linux должны использовать Zabbix agent 2 с активными проверками через назначенный прокси и шифрованием TLS/PSK. Пассивные проверки допускаются только в случаях, когда это требуется типом item или задокументированным операционным исключением.
```

Управление PSK должно определять:

- кто генерирует ключи;
- где хранятся ключи;
- процесс ротации;
- соглашение об именовании PSK identity;
- как отзывать скомпрометированные ключи;
- допустим ли один PSK на хост, на сегмент или на автоматизированный batch.

Безопасный enterprise-baseline предпочитает PSK identity на хост или хотя бы на сегмент. Единый глобальный PSK для всех продакшн-хостов прост, но создаёт большой blast radius.

## Исключения

Допустимые исключения:

| Исключение | Пример | Требуемый контроль |
|---|---|---|
| Требуется пассивная проверка | специфическая кастомная метрика | задокументировать причину на уровне item/шаблона |
| Legacy ОС / приложение | старый Windows/Linux-хост | владелец + дата миграции |
| Ограничение appliance | агент недоступен | путь через SNMP/API/web-проверку |
| Особая политика DMZ | входящий доступ только от прокси | согласование ИБ |
| Временная миграция | старый шаблон всё ещё пассивный | дата истечения |

Каждое исключение должно быть видно в инвентаризации или метаданных тегов, например:

```text
agent_mode=passive-exception
exception_owner=platform
exception_review=2026-09-01
```

---

## Критерии приёмки

- [ ] Пилотные хосты с активным агентом получают конфигурацию и отправляют данные.
- [ ] PSK-шифрование включено и подтверждено в логах.
- [ ] Firewall-правила соответствуют выбранной схеме.
- [ ] Checklist подключения хостов включает режим агента и назначение прокси.
- [ ] Пассивные исключения задокументированы.
- [ ] У NOC есть шаги troubleshooting для ситуации «активные проверки агента не приходят».
- [ ] ИБ согласовал PSK / certificate baseline.

---

## Триггеры пересмотра

Пересмотреть этот ADR, если:

- ИБ обязывает перейти на TLS на сертификатах;
- изменилась поддержка agent 2 на ключевых платформах;
- крупный инцидент был вызван обращением с PSK / ключами;
- пассивные исключения превысили согласованный порог;
- топология прокси значительно изменилась.
