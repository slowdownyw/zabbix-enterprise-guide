# ADR-006: Стратегия платформы дашбордов

Статус: Предложен  
Дата: ГГГГ-ММ-ДД  
Владелец: Владелец мониторинга  
Принимают решение: Архитектор мониторинга, руководитель NOC, представитель CIO, ИБ/АСУ ТП, владелец Grafana/платформы  
Дата пересмотра: после пилота дашбордов  
Область: глобальная стратегия дашбордов и отчётности

---

## Что решаем

Где должны жить операционные дашборды и отчёты: в нативных дашбордах Zabbix, в Grafana, как статические отчёты или в смешанной модели?

---

## Контекст

Дашборды — не украшение. Это операционные продукты с аудиторией, целью, частотой обновления, владельцем и связанным действием.

Разным аудиториям нужны разные представления:

```text
NOC:           что горит прямо сейчас?
Команда сервиса: что не так с моим сервисом?
Тимлид:        где шум и операционный долг?
CIO:           выполняют ли бизнес-сервисы ожидания?
ИБ/АСУ ТП:    что видно без нарушения границ?
```

Нативные дашборды Zabbix близки к данным о проблемах и не требуют дополнительной платформы. Grafana сильнее для визуального дизайна, шаблонизации, multi-source представлений и executive/service-дашбордов. Отчёты могут генерироваться из любой платформы или из экспортированных/API-данных.

---

## Критерии выбора

1. Удобство и доверие NOC.
2. Видимость на уровне сервиса.
3. Потребность в multi-source визуализации.
4. Операционная простота.
5. Разграничение доступа и аудиторий.
6. Автоматизация отчётности.
7. Исключить «dashboard sprawl».

---

## Рассмотренные варианты

### Вариант A — Только нативные дашборды Zabbix

Все дашборды живут в Zabbix.

**Выбирать когда:**

- организация хочет минимум инфраструктуры;
- NOC работает напрямую в UI Zabbix;
- нет потребности в log/BI/multi-source дашбордах;
- нет команды, владеющей Grafana.

**Избегать когда:**

- executive-дашборды требуют красивого представления;
- Grafana уже является стандартом;
- дашборды должны брать данные из Loki/SQL/других источников;
- важны шаблонизация и drill-down.

**Плюсы:**

- нет дополнительной платформы;
- прямой доступ к проблемам и хостам Zabbix;
- проще с правами, если RBAC Zabbix уже настроен;
- хорош для операционных экранов проблем.

**Минусы:**

- менее гибкая визуализация;
- слабый multi-source;
- менее подходит для executive-отчётности;
- дизайн дашборда ограничен виджетами Zabbix.

---

### Вариант B — Grafana для всего

Все дашборды в Grafana; UI Zabbix используется преимущественно для конфигурации и деталей проблем.

**Выбирать когда:**

- Grafana уже является enterprise-стандартом;
- нужны multi-source дашборды;
- команды предпочитают UX Grafana;
- нужны переменные, drill-down и красивый layout.

**Избегать когда:**

- Grafana не эксплуатируется надёжно;
- NOC нуждается в немедленных нативных действиях Zabbix;
- модель доступа не защищена;
- качество datasource Zabbix не проверено.

**Плюсы:**

- сильная визуализация;
- multi-source;
- лучше executive/service-дашборды;
- переиспользуемые паттерны дашбордов.

**Минусы:**

- дополнительная зависимость;
- отдельные вопросы прав/SSO;
- риск расхождения дашбордов с семантикой Zabbix;
- NOC может вынужден переключаться между Grafana и Zabbix.

---

### Вариант C — Смешанная модель по аудитории

Нативный Zabbix для операционных/проблемных представлений, Grafana для сервисных, executive, capacity и multi-source дашбордов.

**Выбирать когда:**

- Zabbix остаётся системой учёта проблем;
- Grafana есть или может быть введена в эксплуатацию;
- аудитория NOC и руководства различаются;
- нужны и операции реального времени, и отчётность.

**Избегать когда:**

- организация не может сопровождать два UI;
- никто не отвечает за согласованность дашбордов;
- права невозможно согласовать.

**Плюсы:**

- прагматичная enterprise-модель;
- NOC остаётся близко к истине Zabbix;
- богатые service/executive-дашборды;
- путь к логам/метрикам/SLA-отчётам в будущем.

**Минусы:**

- требует governance;
- две платформы для поддержки;
- риск несогласованности определений без общей tag schema;
- нужно назначить владельцев дашбордов.

---

### Вариант D — Сначала отчёты, дашборды потом

Генерировать периодические PDF/Markdown/HTML-отчёты и откладывать интерактивные дашборды.

**Выбирать когда:**

- руководство сначала хочет месячную/еженедельную отчётность;
- зрелость операционных дашбордов низкая;
- качество данных не готово для real-time дашбордов;
- security review для дашбордов задерживается.

**Избегать когда:**

- NOC нужна ситуационная осведомлённость в реальном времени;
- инциденты требуют живого drill-down;
- команды уже работают с дашбордов.

**Плюсы:**

- проще социализировать с руководством;
- заставляет определить метрики;
- меньше UI/платформенной работы на старте.

**Минусы:**

- не помогает живым операциям;
- отчёты могут стать ручными, если не автоматизированы;
- запаздывает обратная связь по качеству алертов.

---

## Решение

Рекомендуемый baseline:

> Используйте **смешанную модель по аудитории**. Нативные дашборды Zabbix — для NOC и операций с проблемами. Grafana — для сервисных дашбордов, дашбордов тимлидов, CIO/executive-представлений, capacity и multi-source. Периодические отчёты переиспользуют ту же модель тегов/сервисов.

Это решение опирается на один принцип:

> Дашборды должны читать модель тегов/сервисов. Они не должны изобретать свою скрытую классификацию.

---

## Политика размещения дашбордов

| Тип дашборда | Предпочтительная платформа | Причина |
|---|---|---|
| Активные проблемы NOC | Нативный Zabbix или Grafana с прямым drill-down в проблемы | Должен отражать текущее состояние проблем Zabbix |
| Дашборд по сервису | Grafana | Лучший layout, переменные, multi-source drill-down |
| Дашборд тимлида | Grafana | Тренды, шумные алерты, метрики MTTR/ack |
| CIO-дашборд | Grafana/отчёт | Бизнес-язык, красивое резюме |
| Представление директора / завода | Отчёт/push-сводка | Обычно не живой технический дашборд |
| ИБ-дашборд | Grafana или ИБ-платформа | Multi-source, контроль доступа |
| OT-дашборд | Отдельное / строго ограниченное представление | Должно соблюдать границы OT/SCADA |
| Capacity-отчёт | Grafana/отчёт/SQL-экспорт | Анализ трендов и прогнозирование |

---

## Последствия

### Положительные

- NOC остаётся близко к истине Zabbix о проблемах.
- Сервисные и executive-представления получают лучший UX.
- Есть путь к multi-source в будущем.
- Отчёты могут переиспользовать те же определения.
- Дизайн дашбордов согласуется со слоистой моделью из главы 05.

### Негативные / компромиссы

- Grafana становится поддерживаемой зависимостью.
- Права нужно проектировать дважды или интегрировать с SSO.
- Дашборды могут расходиться, если нет процесса ревью.
- Владельцы должны быть назначены для каждого дашборда.

### Операционные последствия

- У каждого дашборда должен быть владелец.
- Каждый дашборд должен объявлять аудиторию и действие.
- Фильтры дашбордов должны использовать согласованные теги.
- NOC-дашборд должен быть протестирован в реальном/пилотном инциденте.
- Executive-отчёты должны указывать метод измерения и исключения.

---

## Заметки по реализации

У каждого дашборда должны быть метаданные:

```yaml
dashboard:
  id: service-1c-erp
  title: 1C ERP Service Health
  audience: service-team
  owner: it-apps
  refresh: 1m
  primary_action: расследовать деградацию сервиса
  data_sources:
    - zabbix
    - loki
  required_tags:
    - env
    - service
    - owner
    - component
    - scope
```

У каждого отчёта должны быть метаданные:

```yaml
report:
  id: monthly-sla-summary
  audience: cio
  owner: monitoring-owner
  cadence: ежемесячно
  source_of_truth:
    - zabbix_services
    - service_catalog
  exclusions:
    - approved_maintenance
    - force_majeure
```

---

## Исключения

Временные статические дашборды допустимы для пилотов, но не должны превращаться в постоянные скрытые сервисные каталоги.

Статический список хостов допустим, когда:

- дашборд для короткой миграции;
- теги сервисов ещё не готовы;
- у дашборда есть дата истечения.

---

## Критерии приёмки

- [ ] Решение о платформе дашбордов согласовано.
- [ ] NOC-дашборд существует и показывает активные High/Disaster-проблемы.
- [ ] Хотя бы один дашборд по сервису использует теги `service=...`.
- [ ] CIO/executive-дашборд не содержит только low-level CPU/диски.
- [ ] Для каждого дашборда определён владелец.
- [ ] Календарь отчётов существует.
- [ ] Ревью дашбордов включено в еженедельные/ежемесячные операции.

---

## Триггеры пересмотра

Пересмотреть этот ADR, если:

- Grafana стала недоступна или безвладельца;
- NOC перестал использовать дашборды;
- executive-отчёты требуют других определений;
- изменилась tag schema;
- сервисный каталог стал авторитетным в другом месте;
- появился dashboard sprawl.
