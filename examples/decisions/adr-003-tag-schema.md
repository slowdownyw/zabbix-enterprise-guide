# ADR-003: Обязательная схема тегов

Статус: Предложен  
Дата: ГГГГ-ММ-ДД  
Владелец: Владелец мониторинга  
Принимают решение: Архитектор мониторинга, руководитель NOC, владелец ServiceDesk, владельцы сервисов, ИБ/АСУ ТП  
Дата пересмотра: после пилота, затем ежеквартально  
Область: глобально

---

## Что решаем

Какие теги обязательны для продакшн-хостов и событий, и на каком уровне они должны задаваться?

---

## Контекст

Теги — не косметика. В зрелом дизайне Zabbix теги связывают технические объекты мониторинга с операционным смыслом:

```text
хост → сервис → владелец → среда → площадка → тип проблемы → маршрутизация → дашборд → SLA/отчёт
```

Без обязательной tag schema дашборды строятся руками, алерты маршрутизируются по хрупким host groups, сервисные отчёты требуют ручного сопоставления, а ответственность остаётся размытой.

Это решение нужно принять **до** массового развёртывания. Ретроспективно навесить теги на сотни или тысячи хостов технически возможно, но дорого и политически болезненно.

---

## Критерии выбора

1. Единообразная маршрутизация алертов.
2. Сервис-ориентированные дашборды и отчёты.
3. Прозрачное владение.
4. Интеграция с SLA / сервисным каталогом.
5. Разделение RBAC и технической навигации от сервисной семантики.
6. Поддержка тегов в LLD / prototype.
7. Исключить неконтролируемый drift словаря тегов.

---

## Рассмотренные варианты

### Вариант A — Нет обязательных тегов, команды тегируют по своему усмотрению

Каждая команда может добавлять теги, когда считает нужным.

**Выбирать когда:**

- лабораторная среда;
- очень маленький Zabbix;
- нет отчётности на уровне сервисов;
- нет маршрутизации алертов по команде/сервису.

**Избегать когда:**

- enterprise-среда;
- дашборды должны работать между командами;
- NOC нужна маршрутизация;
- планируется SLA / отчётность;
- сервисы охватывают несколько технических групп.

**Плюсы:**

- минимум барьеров на старте;
- не нужно governance;
- команды могут экспериментировать.

**Минусы:**

- хаос нарастает незаметно;
- одна сущность появляется как `service`, `svc`, `application`, `app`;
- дашборды становятся ненадёжными;
- отчёты требуют ручной очистки;
- маршрутизации алертов нельзя доверять.

---

### Вариант B — Только минимальный набор обязательных host-тегов

Требовать небольшой набор тегов хоста, например `env`, `service`, `owner`, `criticality`.

**Выбирать когда:**

- организация начинает стандартизацию;
- зрелость команды низкая или средняя;
- важна скорость развёртывания;
- дашборды / маршрутизация нужны, но стабильный базис важнее полноты;
- дисциплина тегирования items/triggers ещё не готова.

**Избегать когда:**

- сразу нужны дашборды на уровне компонентов;
- сервисные отчёты требуют точной семантики событий с первого дня;
- шаблоны уже имеют зрелую tag-модель.

**Плюсы:**

- прагматично;
- легко объяснить;
- даёт базовые сервисные дашборды и маршрутизацию;
- низкий порог внедрения.

**Минусы:**

- события о проблемах всё ещё могут не содержать деталей о компоненте/scope;
- LLD-сущности семантически слабы, если prototype не тегирован;
- позднее потребуется доработка.

---

### Вариант C — Полная слоистая tag schema

Определить теги хостов, шаблонов, items/prototype, триггеров и сервисов с допустимыми значениями и правилами размещения.

**Выбирать когда:**

- внедряется book-style дизайн;
- маршрутизация алертов, дашборды, SLA и отчёты должны использовать единую семантическую модель;
- шаблоны перерабатываются;
- команда может соблюдать стандарты;
- есть Git/PR-ревью или audit-скрипты.

**Избегать когда:**

- нет владельца governance тегов;
- команды не могут договориться об определениях сервисов/ответственных;
- проект — это короткий технический PoC.

**Плюсы:**

- сильнейшая семантическая модель;
- дашборды и отчёты становятся системными, а не ручными;
- LLD-сгенерированные items/triggers несут полезный смысл;
- маршрутизация событий может быть точной;
- поддерживает будущую автоматизацию.

**Минусы:**

- требует governance;
- требует обучения;
- неправильные значения становятся системными ошибками;
- конфликты по владению сервисами становятся видимыми.

---

## Решение

Рекомендуемый baseline:

> Используйте **слоистую tag schema**. С первого развёртывания сделайте небольшой набор host-тегов обязательным, а для всех новых и переработанных шаблонов задайте стандарты тегирования items/trigger/prototype.

Обязательные теги хоста:

```text
env
criticality
service
owner
location
segment
os_family
```

Рекомендуемые расширения (хост/роль):

```text
role
backup
sla
pii
kii
vendor
```

Рекомендуемые теги семантики событий:

```text
component
scope
impact
notification
runbook
```

Эталонный словарь значений хранится в [`../tag-schema.yaml`](../tag-schema.yaml).

---

## Правила размещения тегов

### Теги хоста отвечают на вопрос: «кто/что это за объект?»

Примеры:

```text
env=prod
service=1c-erp
owner=it-apps
criticality=P1
location=dc-main
segment=it
os_family=windows
role=1c-app
```

### Теги items/prototype отвечают на вопрос: «что это за метрика/сущность?»

Примеры:

```text
component=filesystem
scope=capacity
fs={#FSNAME}
```

```text
component=network-interface
scope=availability
interface={#IFNAME}
```

### Теги триггеров отвечают на вопрос: «что произошло и что это значит?»

Примеры:

```text
component=rphost
scope=availability
impact=service-degradation
notification=active
runbook=rb-1c-rphost-down
```

### Теги сервисов отвечают на вопрос: «как это влияет на бизнес-дашборды?»

Примеры:

```text
service=1c-erp
sla=business-hours
criticality=P1
```

## Последствия

### Положительные

- NOC может фильтровать по `env`, `service`, `criticality`, `owner`.
- Дашборды можно генерировать по сервису/владельцу/площадке.
- SLA-отчёты могут отображать проблемы на сервисы.
- Маршрутизация алертов базируется на метаданных, а не на соглашениях об именовании хостов.
- LLD-items становятся полезными в дашбордах, если prototype тегирован.

### Негативные / компромиссы

- Словарь тегов нужно контролировать.
- Существующие хосты нужно аудировать и привести в порядок.
- Конфликты по владению сервисами больше нельзя скрыть.
- Автоматизация должна аккуратно мержить теги: обновление хоста может затереть существующие теги при небрежной реализации.

### Операционные последствия

- Подключение нового хоста должно включать обязательные теги.
- Ревью шаблонов должно включать ревью тегов.
- Дизайн дашборда должен использовать tag schema, а не изобретать локальные фильтры.
- Еженедельный review алертов должен выявлять отсутствующие / неправильные теги.
- Изменения tag schema требуют PR/ревью или эквивалентного согласования.

---

## Заметки по реализации

1. Начните с `examples/tag-schema.yaml`.
2. Выгрузите текущие хосты и проаудируйте отсутствующие обязательные теги.
3. Заполните недостающие теги хостов через CSV/workshop с владельцами сервисов.
4. Применяйте теги через API аккуратно: мержите существующие теги, не перезаписывайте вслепую.
5. Для новых шаблонов требуйте ревью тегирования items/prototype/trigger.
6. В дашбордах используйте фильтры по тегам, а не хардкод-списки хостов.

Рекомендуемые измерения для аудита:

```text
хосты без env
хосты без service
хосты без owner
хосты без criticality
хосты с неизвестным значением service
хосты с service, которого нет в сервисном каталоге
триггеры High+ без тега runbook или URL
LLD prototype без тегов component/scope
```

---

## Исключения

Допустимые исключения:

- lab/test-хосты вне scope;
- транзитные discovery-объекты, ещё не классифицированные;
- инфраструктурные компоненты, которые не отображаются на бизнес-сервис — должны использовать `service=platform-monitoring`, `service=network-core` или другое согласованное значение, не пустое.

Пустые обязательные теги — не допустимое долгосрочное состояние.

---

## Критерии приёмки

- [ ] Список обязательных тегов согласован.
- [ ] Допустимые значения определены в `tag-schema.yaml`.
- [ ] Пилотные хосты имеют обязательные теги.
- [ ] Проблемы наследуют полезные теги из хоста/шаблона/item/триггера.
- [ ] NOC-дашборд может фильтровать по `env=prod` и `severity>=High`.
- [ ] Хотя бы один сервисный дашборд использует `service=...` вместо статического списка хостов.
- [ ] Действие алерта может маршрутизировать по `owner` или `service`.
- [ ] Есть аудит отсутствующих тегов.

---

## Триггеры пересмотра

Пересмотреть этот ADR, если:

- сервисный каталог значительно изменился;
- дашборды требуют полей, которых нет в tag schema;
- маршрутизация алертов стала слишком сложной;
- команды создают дублирующиеся ключи/значения тегов;
- крупная реорганизация меняет модель ответственности.
