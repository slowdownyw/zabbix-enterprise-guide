# Architecture Decision Records для внедрения Zabbix

Статус: 🟢 концептуально стабильно / 🟡 адаптируйте под свою организацию

Эта директория содержит **Architecture Decision Records (ADR)** для enterprise-внедрения мониторинга на Zabbix.

Цель — не бюрократия. Цель — не дать проекту молча скатиться к неявным решениям:

- «Почему у нас пять прокси?»
- «Почему агенты активные по умолчанию?»
- «Почему у каждого хоста должны быть теги `service` и `owner`?»
- «Почему дашборды в Grafana, а NOC работает в Zabbix?»
- «Кто разрешил вносить изменения в шаблоны напрямую в UI?»

Если эти решения не записаны, они всё равно принимаются — только неформально, непоследовательно, и обычно тем, кто говорит громче всех.

---

## Что такое ADR

**ADR** — небольшой документ, фиксирующий одно значимое решение:

```text
Контекст → Варианты → Решение → Последствия → Критерии пересмотра
```

Это не проектный документ и не протокол совещания. Он отвечает на один вопрос:

> «Учитывая текущий контекст, что мы выбрали, почему, и что это означает?»

ADR особенно полезен для проектов мониторинга, потому что многие решения выглядят технически, но на самом деле операционны:

- топология прокси влияет на сетевое владение и firewall-политику;
- tag schema влияет на дашборды, SLA, маршрутизацию алертов, отчётность и RBAC;
- политика retention влияет на размер БД, бюджет хранилища и глубину troubleshooting;
- GitOps-политика определяет, кто может менять продакшн-мониторинг и как работает откат;
- платформа дашбордов определяет, кто видит операционную правду и где команды работают каждый день.

---

## Когда создавать ADR

Создавайте ADR, когда решение:

1. **Сложно обратимо** — изменить его позже дорого или политически болезненно.
2. **Межкомандное** — затрагивает более одной команды.
3. **Операционно видимо** — меняет алертинг, дашборды, SLA, доступ или incident-workflow.
4. **Архитектурообразующее** — влияет на остальную модель мониторинга.
5. **Дискуссионное** — есть несколько разумных вариантов.

**Не** создавайте ADR для мелких деталей реализации:

```text
❌ точный цвет панели в Grafana
❌ исправление опечатки в названии триггера
❌ разовое временное maintenance-окно
❌ значение локального макроса для одного хоста
```

Создавайте ADR для таких решений, как:

```text
✅ активный vs пассивный режим агента по умолчанию
✅ топология server/proxy
✅ обязательная tag schema
✅ иерархия host groups
✅ severity-политика
✅ baseline retention и sizing
✅ стратегия платформы дашбордов
✅ модель change control для шаблонов
✅ граница мониторинга OT/SCADA
```

## Жизненный цикл ADR

Рекомендуемые статусы:

| Статус | Значение |
|---|---|
| `Предложен` | Черновик решения, всё ещё обсуждается |
| `Принят` | Согласован и активен |
| `Заменён` | Заменён другим ADR |
| `Устарел` | Исторически верен, но не должен использоваться для новой работы |
| `Отклонён` | Рассмотрен и явно не выбран |

Документы ADR должны быть неизменны по духу. Если решение значительно изменилось — не переписывайте историю. Создайте новый ADR и пометьте старый как `Заменён`.

Пример:

```text
ADR-002: По умолчанию активные агенты с PSK — Принят
ADR-017: Перевод хостов DMZ на пассивные проверки через прокси — Заменяет ADR-002 для области DMZ
```

## Кто владеет ADR

Владение ADR должно быть явным.

| Роль | Ответственность |
|---|---|
| Владелец мониторинга | Обеспечивает фиксацию и пересмотр решения |
| Архитектор мониторинга | Готовит варианты и рекомендацию |
| Администратор Zabbix | Проверяет реализуемость в Zabbix |
| Владелец платформы/БД | Проверяет последствия для сервера, БД, хранилища, backup |
| Сетевой инженер | Проверяет размещение прокси, SNMP, firewall и маршрутизацию |
| ИБ/АСУ ТП | Проверяет шифрование, границы доступа, ограничения OT/КИИ |
| Владелец приложения | Проверяет влияние на сервис и runbook'и |
| Руководитель ServiceDesk/NOC | Проверяет операционный workflow и последствия для эскалации |
| Спонсор/CIO | Согласовывает решения с бюджетным, политическим или межкомандным влиянием |

Один ADR не требует всех ролей. Но если роль затронута и не проконсультирована — решение хрупкое.

## Как пользоваться этой директорией

Начните так:

1. Скопируйте [`adr-template.md`](./adr-template.md).
2. Присвойте следующий номер: `adr-009-...md`.
3. Назовите кратко и по сути решения.
4. Пишите варианты честно. Не создавайте фиктивные варианты только для обоснования выбранного.
5. Ссылайтесь на соответствующие главы руководства.
6. Фиксируйте последствия, особенно негативные.
7. Добавляйте критерии пересмотра: когда нужно вернуться к этому решению?

Соглашение об именовании:

```text
adr-001-deployment-model.md
adr-002-agent-mode-and-encryption.md
adr-003-tag-schema.md
```

## Рекомендуемая последовательность решений

В реальном внедрении решения должны приниматься примерно в таком порядке:

### Фаза 0 — Мобилизация

```text
ADR-001: Модель развёртывания
ADR-002: Топология прокси
ADR-003: Режим агента и baseline шифрования
ADR-004: Baseline sizing и retention
```

Эти решения влияют на инфраструктуру, firewall-правила, sizing VM, БД и первоначальные заявки на доступ. Их нельзя откладывать до разработки шаблонов.

### Фаза 1 — Discovery

```text
ADR-005: Источник истины во время внедрения
ADR-006: Обработка и конфиденциальность discovery-данных
ADR-007: Начальный scope сервисного каталога
```

Эти решения не дают discovery превратиться в неконтролируемую свалку данных.

### Фаза 2 — Дизайн и стандартизация

```text
ADR-008: Обязательная tag schema
ADR-009: Иерархия host groups
ADR-010: Модель композиции шаблонов
ADR-011: Severity-модель
ADR-012: LLD-политика
ADR-013: Политика покрытия runbook'ами
```

Эти решения определяют, как создаются и поддерживаются объекты Zabbix.

### Фаза 3 — Пилот и развёртывание

```text
ADR-014: Стратегия платформы дашбордов
ADR-015: Модель маршрутизации и эскалации алертов
ADR-016: Модель change control для шаблонов
ADR-017: Политика maintenance-окон
```

Эти решения влияют на повседневные операции.

### Фаза 4 — Передача

```text
ADR-018: Модель владения мониторингом
ADR-019: Процесс еженедельного ревью алертов
ADR-020: Владение backlog'ом после проекта
```

Эти решения поддерживают систему в живом состоянии после завершения проекта внедрения.

## Минимальное правило

Если вы принимаете только одно правило, примите это:

> Любое решение, меняющее маршрутизацию алертов, ответственность, семантику дашбордов, отчётность о доступности сервисов, границы безопасности или workflow изменений в продакшне, должно быть зафиксировано как ADR.
