# Терминологическая политика книги

Справочник для редактуры и аудита терминологии. Определяет, что заменять в тексте,
а что оставлять как устоявшийся термин.

**Это источник истины для редактуры.** Если возник вопрос «как написать» — сначала ищем здесь, потом в [glossary.md](glossary.md), и только если ничего не нашли — принимаем новое решение и **фиксируем его сюда же**, чтобы не плыть.

При сомнениях по терминам сторонних продуктов и стандартов — сверять с русской документацией соответствующего вендора (Microsoft Learn, Zabbix docs ru, Positive Technologies, ГОСТ).

---

## Базовые правила формы

### Кавычки

- В русском тексте — ёлочки `«…»`.
- Внутри ёлочек для вложенной цитаты — лапки `„…"`.
- В коде, mermaid-диаграммах, iframe-атрибутах, HTML — стандартные двойные `"…"`. Это синтаксис, не текст.
- Прямые двойные `"…"` в обычном русском тексте **запрещены**.

### Апостроф при склонении англицизмов

- `runbook'и`, `runbook'ами`, `runbook'ом`, `host'ы` — **запрещены** во всех падежах.
- Варианты: оставлять без склонения («без runbook», «ссылка на runbook», «runbook есть/нет»), либо заменять русским словом («инструкция», «регламент»).

### Статусные слова Zabbix

- `PROBLEM`, `OK`, `RESOLVED` — прописными буквами как технические маркеры состояния, не существительные.
- В тексте: «событие со статусом PROBLEM», «триггер перешёл в OK», «событие RESOLVED».
- **Не** «PROBLEM event», **не** «problem event» в основном тексте — переводить как «событие PROBLEM» или «проблемное событие» (последнее ближе к Zabbix RU). В коде, таблицах и именах категорий UI (`event tags`) допустимо как имя.

### Гибридные дефисные конструкции `<English>-<русское>`

Гибрид допустим, если **левая часть — имя собственное (продукт, инструмент, протокол, аббревиатура)**, и конструкция встречается в русскоязычной ИТ-литературе или в документации вендора:

- ✅ Zabbix-сервер, Zabbix-агент, Zabbix-прокси
- ✅ API-вызов, HTTP-запрос, TLS-сертификат
- ✅ Git-репозиторий, GitOps-подход
- ✅ SNMP-опрос, IPMI-канал
- ✅ DR-площадка, SLA-отчёт, P1-инцидент

Гибрид **запрещён**, если левая часть — обычное английское прилагательное (Base, Role, Application, Synthetic, Linked, Community, Backup и т.п.):

- ❌ Base-шаблон → ✅ «OS Base template» / «базовый шаблон»
- ❌ Role-шаблон → ✅ «Role template» / «ролевой шаблон»
- ❌ Application-шаблон → ✅ «Application template» / «прикладной шаблон»
- ❌ Synthetic-шаблон → ✅ «Synthetic template» / «синтетический шаблон»
- ❌ Community-шаблоны → ✅ «шаблоны сообщества» / «Community templates»
- ❌ Backup-задача → ✅ «задача резервного копирования»

**Отдельный случай — «X группы Y».** Если речь о принадлежности (член/участник/узел/сервер группы), правильная форма — через родительный падеж: **«член DAG»**, «узел кластера», «сервер группы доступности», не «DAG-член», «cluster-узел». Это согласуется с русской документацией Microsoft Exchange ([Microsoft Learn](https://learn.microsoft.com/ru-ru/exchange/high-availability/disaster-recovery/recover-dag-member-servers)).

**Признак для проверки:** если левая часть — обычное английское прилагательное-определение, не имя продукта/протокола — гибрид запрещён.

### Расшифровка аббревиатур

- При первом упоминании в тексте — **только если** термина нет в [glossary.md](glossary.md).
- Если термин в глоссарии — не дублировать расшифровку в основном тексте; ссылка на главу/глоссарий через гиперссылку.

| trigger | триггер | триггер |

### Теги Zabbix в коде

Значения тегов `owner=`, `scope=`, `service=`, `env=` в кодовых блоках, YAML,
примерах конфигурации — **не трогаем**. Это технические идентификаторы, а не
художественный текст.

## Термины, которые заменяем в тексте

### Заменить обязательно

Замена не меняет смысл, есть устоявшийся русский эквивалент.

| Английский термин / калька | Правильный русский вариант |
|---|---|
| root cause | первопричина |
| root cause analysis / RCA | анализ первопричин (RCA) |
| root cause finding | поиск первопричины |
| корневая причина | первопричина (это калька, не устоявшийся термин) |
| action item | задача / пункт плана (PMBOK RU: «пункт действий») |
| stakeholder | заинтересованная сторона (PMBOK RU, ГОСТ Р ИСО/МЭК 12207) |
| handoff / handover | передача / передача в эксплуатацию (PMBOK RU: «передача дел») |
| quick win | быстрое улучшение |
| freeze / freeze period | заморозка изменений / период запрета изменений |
| freeze calendar | календарь заморозки / периоды запрета изменений |
| noisy alert / noisy trigger | шумный алерт / шумный триггер |
| alert fatigue | усталость от алертов |
| onboarding хоста | ввод хоста в мониторинг / добавление хоста |
| incident flow | жизненный цикл инцидента / порядок обработки инцидента |
| эвент | событие (для `trigger`, `event`, `problem event` см. таблицу официальных терминов Zabbix выше) |
| переиспользоваться / переиспользование | применяться повторно / повторное использование |
| прилинковать / залинковать / линкование / привязать шаблон / привязка шаблонов | присоединить / присоединение (официальный термин Zabbix RU: [docs/templates/linking](https://www.zabbix.com/documentation/current/ru/manual/config/templates/linking)) |
| linked template | связанный шаблон / присоединённый шаблон («linked template» оставляем в кодовых конструкциях UI) |
| апгрейд | обновление |
| инстанция (в смысле instance) | экземпляр |
| middleware (применительно к СУБД) | СУБД / реляционная БД. MSSQL, PostgreSQL и т.п. — это СУБД, а не middleware. Middleware — это слой между приложением и БД: серверы приложений, очереди, ESB. |

### Заменить желательно

Контекстуально: смотреть, не является ли слово частью устоявшегося термина.

| Английский термин | Предпочтительный русский вариант | Когда оставить |
|---|---|---|
| owner (в тексте) | ответственный / владелец сервиса / команда-владелец | Только в значении тега `owner=` в коде |
| ownership | ответственность / владение | Когда это не термин из GitOps/RBAC |
| baseline | базовый уровень / исходное состояние (ГОСТ: «базовая линия» / «базовое состояние») | Если это не «baseline configuration» как термин |
| gap | пробел / расхождение / зона непокрытия | Если не в составе «gap analysis» как методология |
| scope (в тексте) | область / охват / область работ | Только в значении тега `scope=` в коде |
| review (существительное в тексте) | разбор / обзор / ревизия / ретроспектива | Если не «code review», «peer review» как термины практики |
| capacity planning | планирование ёмкости / планирование мощностей (ITIL RU: «управление мощностями») | — |
| on-call | дежурный / дежурная смена / дежурство (ITIL RU: «дежурство по вызову») | Если не в названии инструмента (PagerDuty on-call) |
| escalation path | путь эскалации / порядок эскалации | Само слово «эскалация» оставляем |

### Нужно решение автора

Неоднозначные случаи — зависит от контекста и желаемого стиля.

| Термин | Варианты | Вопрос |
|---|---|---|
| postmortem (в обычном тексте) | разбор инцидента / постмортем | Если термин уже объяснён в главе — оставить; в проходных упоминаниях — подумать |
| SRE-практики / SRE-подход | практики надёжности | Сокращение SRE устоялось, но «SRE-практики» звучит как жаргон |
| blue-green / canary | сине-зелёный / канарочный | Оба варианта встречаются в русской литературе |
| error budget | бюджет ошибок | Термин SRE, объясняется в книге |
| runbook (вне главы про runbooks) | инструкция / регламент действий | Если это не ссылка на концепцию runbook — подумать |

## Что не трогаем никогда

- Кодовые блоки, команды, конфигурации (YAML, JSON, Python, Bash, SQL)
- URL, имена файлов, пути
- Названия API-методов и параметров Zabbix (`configuration.export`, `auditlog.get`)
- Значения тегов и макросов в примерах (`owner=it-apps`, `{$THRESHOLD}`)
- Цитаты и названия внешних документов/стандартов
- Имена собственные: продукты, компании, авторы
